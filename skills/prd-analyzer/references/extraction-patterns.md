# PRD 提取信号

从原始 PRD 中提炼 5 个 zone 的信息，不受 PRD 格式和结构的限制。

## 核心原则

PRD 的格式千差万别——有的是结构化文档，有的是自由文本叙述，有的是用户故事集合，有的只是一段对话记录。但无论格式如何，我们需要提取的信息类型是固定的：5 个 zone。

不关注 PRD 的章节结构，只关注内容中的**信号**——那些指示有价值信息的词语、句式和模式。

## 提取信号

### Zone 1：功能识别

**要找什么**：独立的、可实施的功能单元。

**信号词**：
- "功能"、"支持"、"提供"、"实现"、"开发"、"新增"、"添加"
- "需要能"、"应该能"、"可以"、"must"、"should"、"able to"
- 编号列表中的需求项
- 标题中包含"功能"、"需求"、"Feature"、"Requirement"的章节内容

**判断标准**：
- 每个功能必须是独立可测试的
- 如果一段描述涉及多个不相关的行为，拆分为多个功能
- 功能粒度参考：一个功能 = 一个用户可以完成的端到端操作，或一个独立的技术能力

### Zone 2：用户故事和场景

**要找什么**：谁、在什么情况下、做什么、得到什么结果。

**信号词**：
- "作为...我想要..."、"As a...I want..."
- "用户"、"管理员"、"系统"、"第三方"、"运营"、"客户"
- "当...时"、"如果...则"、"在...情况下"
- "流程"、"步骤"、"操作"、"交互"、"点击"、"提交"、"输入"
- "成功"、"失败"、"错误"、"异常"、"超时"

**判断标准**：
- 每个场景必须包含：谁做 + 做什么 + 结果是什么
- 正常流程、边界情况、错误条件都要找齐
- 如果 PRD 只描述了正常流程，将缺失的错误和边界情况记入歧义列表

### Zone 3：业务实体与交互需求

**要找什么**：PRD 中谈论的业务对象、它们的属性和关系、以及系统需要支持的交互。

**实体信号词**：
- 反复出现的名词（用户、订单、商品、支付、库存）
- "管理"、"包含"、"属于"、"关联"等关系词连接的对象
- 表格、列表中的对象名称
- "有以下属性"、"包含"、"由...组成"

**属性信号词**：
- 跟在实体后面的描述（"订单有金额、状态、创建时间"）
- "名称"、"编号"、"金额"、"时间"、"状态"、"类型"等常见属性词
- "必填"、"可选"、"唯一"、"默认"等约束性描述

**关系信号词**：
- "一个...多个"、"属于"、"包含"、"关联"、"引用"
- "一对多"、"多对多"、"唯一"
- 实体之间通过动词连接（"用户**下单**"、"订单**包含**商品"）

**交互需求信号词**：
- "创建"、"查询"、"更新"、"删除"、"搜索"、"筛选"、"排序"、"导出"、"导入"
- "提交"、"审批"、"发布"、"审核"、"确认"、"取消"
- "需要校验"、"需要检查"、"需要通知"

**状态流转信号词**：
- "待... → 已..."、"初始"、"完成"、"过期"、"取消"
- "状态变更"、"流程"、"审批流"、"工单流转"
- 用箭头、破折号或序号表示的变化序列

**注意**：只记录 PRD 中的业务描述。如果 PRD 中提到"订单金额"，记录"金额"作为属性，但不推导为 `decimal(10,2)`。如果 PRD 中提到"查询订单"，记录为交互需求，但不推导为 `GET /api/orders`。

### Zone 4：约束条件

**要找什么**：系统必须满足的非功能性要求。

**性能信号**：
- "响应时间"、"延迟"、"毫秒"、"秒"、"QPS"、"TPS"、"并发"
- "不能超过"、"必须在...以内"、"至少"、"最多"
- "快速"、"大量"、"高性能"（模糊量词，记入歧义）

**安全信号**：
- "登录"、"认证"、"授权"、"权限"、"角色"
- "加密"、"脱敏"、"隐私"、"敏感信息"
- "审计"、"日志"、"追踪"、"合规"
- "HTTPS"、"Token"、"OAuth"、"SSO"

**兼容性信号**：
- "浏览器"、"设备"、"移动端"、"PC端"、"版本"
- "向下兼容"、"向后兼容"、"API版本"
- "iOS"、"Android"、"Chrome"、"Safari"

**业务规则信号**：
- "如果...则...必须"、"不允许"、"限制"、"规则"
- "计算公式"、"折扣"、"费率"、"阈值"
- "唯一"、"不可重复"、"校验"

**合规信号**：
- "法规"、"监管"、"GDPR"、"个人信息保护"
- "数据保留"、"数据删除"、"数据导出"
- "备案"、"审计"、"上报"

### Zone 5：验收条件

**要找什么**：判断功能是否完成的可测试标准。

**信号词**：
- "验收标准"、"Acceptance Criteria"、"Definition of Done"、"DoD"
- "应该显示"、"必须返回"、"不能超过"、"确保"
- "测试"、"验证"、"检查"、"确认"
- "通过条件"、"成功标准"、"完成条件"

**判断标准**：
- 每个条件必须是布尔型（通过/不通过）
- 如果 PRD 中的条件模糊（如"体验流畅"），记入歧义并建议具体化
- 每个 Zone 1 的功能都应该有对应的验收条件
- 验收条件应覆盖正常路径、边界情况和错误路径

## 歧义信号

以下模式通常指示 PRD 中的信息不完整，需要记入歧义列表：

| 模式 | 示例 | 应对 |
|------|------|------|
| 模糊量词 | "快速响应"、"大量数据"、"很多用户" | 请求具体数字 |
| 缺失错误处理 | 只描述了成功流程 | 询问失败场景 |
| 未定义术语 | 领域行话没有解释 | 请求定义 |
| 矛盾描述 | 两处说法不一致 | 询问以哪个为准 |
| 缺失边界 | 没提空列表、零结果、最大值 | 标记缺失 |
