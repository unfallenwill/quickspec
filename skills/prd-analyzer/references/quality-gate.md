# 质量门禁：PRD 分析

## 产出物

`${ARGUMENTS}/prd-analysis.md`

## 目标

将非结构化的 PRD 转化为结构化需求，消除歧义，建立后续步骤所需的统一事实来源。

## 使用场景

| 消费者 | 消费什么 |
|--------|----------|
| Step 3 (代码库映射) | 功能关键词和业务实体名称，用于代码库搜索定位 |
| Step 4 (生成规格) | 功能列表、业务实体与交互需求、约束条件，作为 spec 的输入事实；由 spec-creator 结合代码库映射推导技术规格 |
| Step 5 (质量审查) | 功能列表和验收条件，用于交叉对照 PRD 覆盖度 |

## 验收标准

### 文件存在性

- [ ] `${ARGUMENTS}/prd-analysis.md` 文件存在且可读
- [ ] 文件非空（内容长度 > 0）
- [ ] 文件结构正确（包含 5-zone 章节结构）

### 完整性

- [ ] 5 个 zone 全部填充（Feature / User Story / 业务实体与交互需求 / Constraint / Acceptance Criteria）
- [ ] 每个 Feature 有唯一编号（F1, F2...）和优先级
- [ ] 每个 Feature 有关联功能的引用（如有）

### 准确性

- [ ] PRD 中的每个需求点至少映射到一个 zone 的条目（覆盖率检查）
- [ ] 用户故事包含角色、操作和收益三要素
- [ ] 业务实体已识别并标注属性（业务语言，非技术类型）
- [ ] 实体之间的关系已标注
- [ ] 交互需求已提取并关联到涉及的实体
- [ ] 约束条件已分类（性能/安全/兼容性/业务/合规）
- [ ] 区域 3 不包含技术规格内容（如字段类型、API 方法/路径）— 技术规格由 spec-creator 生成

### 返回状态

- [ ] 返回状态包含 `[FEATURE-NAME]` 且非空
- [ ] 返回状态包含 `[AMBIGUITY-COUNT]`，值为歧义章节中的条目数

### 歧义处理

- [ ] 产出文件包含"歧义（需确认）"章节
- [ ] 每个歧义已编号并标注类型（模糊量词 / 缺失错误处理 / 未定义术语 / 矛盾需求 / 缺失边界）
- [ ] 每个歧义以具体问题形式呈现（非模糊担忧）
- [ ] 返回状态的 [SUMMARY] 包含歧义数量提示
- [ ] 歧义由编排器在检查点展示给用户确认（非本 skill 职责）

### 用户确认

- [ ] 用户确认分析结果符合 PRD 原意（检查点通过）
